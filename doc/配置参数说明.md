# 配置参数说明

本文档用于补充 `README.md` 中较为简洁的配置说明，帮助你快速理解 `config/config.yaml` 中各参数的用途。

需要特别说明：

- 当前运行行为仍以 `config/config.yaml`、`train.py`、`predict.py` 和 `src/` 中实际代码为准。
- `README.md` 更偏向“当前真实可运行行为”的说明；本文档更偏向“配置参数释义”和“阅读辅助”。
- 如果本文档与运行结果出现冲突，应优先以 `README.md` 和实际代码行为为准，再回过头修正文档。

## 1. `data` 段

### `file_path`

- 训练数据 CSV 文件路径。
- `train.py` 会从这里读取原始样本数据。

### `target_column`

- 目标列名称，也就是模型要预测的标签列。
- 当前项目默认使用 `Nexp (kN)`，表示试验极限承载力。

### `target_transform.enabled`

- 是否在训练前对目标列做变换。
- 关闭时，模型直接在原始物理量空间拟合目标值。

### `target_transform.type`

- 目标变换类型。
- 当前代码路径支持 `log`、`sqrt` 或不变换。
- 其中：
  - `log` 常用于压缩长尾标签分布；
  - `sqrt` 当前仓库中仍属于不建议启用的路径，详见 `README.md` 的“已知限制”。

### `columns_to_drop`

- 训练前要剔除的列。
- 当前配置里主要用于移除绝对几何量，保留更适合建模的特征表达。

### `test_size`

- 测试集划分比例。
- `train.py` 会先按这个比例做 `train/test` 划分，再在训练部分继续做验证集划分。

### `random_state`

- 数据划分时使用的随机种子。
- 主要影响 `train/test` 和 `train/validation` 的可复现性。

## 2. `model.params` 段

这一段是当前项目中 **XGBoost 超参数的唯一合法入口**。`train.py` 会把它直接整理后传给 `xgb.XGBRegressor`。

### `objective`

- 学习目标函数。
- 当前项目使用 `reg:squarederror`，表示回归任务下的平方误差目标。

### `max_depth`

- 每棵树的最大深度。
- 数值越大，模型表达能力通常越强，但也更容易过拟合。

### `learning_rate`

- 学习率，也可理解为每轮 boosting 的步长。
- 一般越小越稳，但通常需要配合更多树数使用。

### `n_estimators`

- 基学习器数量，也就是 boosting 轮数 / 树的数量。
- 它和 `learning_rate` 往往需要配合看。

### `subsample`

- 每棵树训练时抽取的样本比例。
- 小于 `1.0` 时可引入随机性，有助于抑制过拟合。

### `colsample_bytree`

- 每棵树训练时抽取的特征比例。
- 小于 `1.0` 时表示并不是每棵树都使用全部特征。

### `min_child_weight`

- 子节点最小权重约束。
- 数值更大时，模型通常会更保守一些。

### `reg_alpha`

- `L1` 正则项系数。
- 用于增强稀疏性、抑制复杂度。

### `reg_lambda`

- `L2` 正则项系数。
- 用于平滑参数、抑制模型过拟合。

### `gamma`

- 节点继续分裂所需的最小损失下降量。
- 值越大，分裂会更保守。

### `random_state`

- XGBoost 模型内部使用的随机种子。
- 它影响模型训练过程中的随机性，但不等同于 `cv.random_state`。

### `tree_method`

- 树构建算法。
- 当前配置使用 `hist`，属于较常见、效率较高的实现路径。

### `device`

- 训练设备。
- 当前默认配置为 `cpu`，避免 GPU 依赖。

### `n_jobs`

- 并行线程数。
- `-1` 表示尽量使用可用 CPU 资源。

## 3. `model` 段（训练控制参数）

### `use_optuna`

- 是否在正式训练前先执行 `Optuna` 超参数搜索。
- 开启后，项目会先调参，再用搜索到的最优参数训练最终模型。

### `n_trials`

- `Optuna` 最大试验次数。
- 次数越多，搜索范围通常越充分，但训练成本也更高。

### `optuna_timeout`

- `Optuna` 允许的最长搜索时间，单位为秒。
- 当试验数和时间同时存在时，通常哪个先到就先停。

### `optuna_storage_url`

- `Optuna study` 的持久化存储位置。
- 当前默认指向 SQLite，便于断点续跑和复用历史试验。

### `best_params_path`

- 保存最优参数快照的 JSON 路径。
- 当前仓库还会结合 `context_hash` 判断这份参数是否适用于当前数据上下文。

### `early_stopping_rounds`

- 早停轮数。
- 只有在训练阶段实际提供了验证集 / `eval_set` 时才会生效。

### `eval_metric`

- 训练时监控的验证指标。
- 当前配置使用 `rmse`。

### `validation_size`

- 从训练部分再切出验证集的比例。
- 设为 `0` 时表示不再单独划分验证集。

## 4. `cv` 段

这一段用于定义交叉验证协议。

### `n_splits`

- 交叉验证折数。
- 当前代码会在 `Optuna` 调参和训练阶段输出的交叉验证报告中共同使用这个折数。

### `shuffle`

- 是否在划分各折之前打乱样本。
- 如果关闭，则样本按原始顺序分折。

### `random_state`

- 交叉验证折分所使用的随机种子。
- 当前代码在 `shuffle: true` 时会用它保证折分可复现。

## 5. `paths` 段

这一段主要描述项目的输出路径和命名约定。

需要注意：

- `output_dir` 会直接影响默认输出目录；
- 其余键更多承担项目配置和命名约定的说明作用；
- 实际生成文件名请优先以 `README.md` 中的“训练输出”一节和真实产物为准。

### 常见字段

- `output_dir`：默认输出目录。
- `model_dir`：模型目录约定。
- `model_output`：模型文件名约定。
- `preprocessor_output`：预处理器文件名约定。
- `feature_names_output`：特征名文件名约定。
- `metadata_output`：训练元数据文件名约定。
- `evaluation_report`：评估报告文件名约定。
- `logs_dir`：日志目录约定。

## 6. `logging` 段

这一段用于描述日志相关配置项，包括：

- `level`：日志等级。
- `format`：日志格式模板。
- `file_name`：日志文件名。

如果你后续继续扩展日志初始化逻辑，这一段就是统一的配置入口。

## 7. 建议如何阅读这份配置

如果你要从工程角度快速理解当前项目，建议按下面顺序看：

1. 先看 `data`：确认数据来源、标签列和是否启用目标变换。
2. 再看 `model.params`：确认 XGBoost 的实际建模超参数。
3. 再看 `model`：确认是否启用 `Optuna`、是否使用验证集和早停。
4. 再看 `cv`：确认调参与训练期 CV 报告的实验协议。
5. 最后看 `paths`：确认模型、报告和图表的输出位置。
